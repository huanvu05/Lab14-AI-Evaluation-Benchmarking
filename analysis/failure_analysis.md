# Failure Analysis Report

## 1. Failure Clustering
During the evaluation of V2 Optimized, we observed the following issues across tests:
- **Total Test Cases**: 50
- **Low Score Cases (< 5)**: 50
- **Judge Conflict Cases**: 49

### Key Clusters:
1. **Retrieval Failure**: Instances where Hit Rate = 0.
2. **Hallucination**: High context overlap but answer length excessive, resulting in penalties from the GPT judge.
3. **Context Mismatch**: Expected retrieval IDs did not match retrieved.

## 2. 5 WHYS Analysis

### Cluster 1: Multi-Judge Disagreement (Conflicts)
**Problem**: The GPT Judge and Heuristic Judge score differ significantly.
1. **Why?** The Heuristic Judge penalizes simple answers strictly based on exact keyword overlap, while GPT Judge accepts semantic meaning.
2. **Why?** The rule-based threshold for overlap was set too rigidly.
3. **Why?** Semantic synonyms used by the generative LLM aren't caught by basic set intersection in rules.
4. **Why?** The evaluator is a rigid script without embedding-based similarity comparison.
5. **Why?** (Root Cause) The secondary judge lacks robust semantic understanding, causing spurious penalization for valid algorithmic paraphrasing. 
*Fix Action*: Upgrade Heuristic Judge to use cosine similarity on embeddings rather than exact keyword overlap.

### Cluster 2: Occasional Low Retrieval Score
**Problem**: Hit rate isn't 100% on some tricky questions.
1. **Why?** The Agent retrieves sources but misses the exact expected document ID.
2. **Why?** The search query generated by the agent is too verbatim to the confused user prompt.
3. **Why?** The user query lacks entity-specific keywords (e.g. asking "Làm thế nào..." without context).
4. **Why?** Context missing in the conversation state prior to search.
5. **Why?** (Root Cause) The system lacks a query rewriter step prior to retrieval.
*Fix Action*: Implement step-back prompting or query rewriting in RAG pipeline.
